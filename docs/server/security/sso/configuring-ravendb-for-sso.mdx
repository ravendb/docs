# SSO: Configuring RavenDB

<Admonition type="note" title="">

Once the [SSO application](../../../server/security/sso/deploying-sso-app.mdx) is deployed, a RavenDB cluster needs two things to
accept users coming through it:

1. The **SSO server's public certificate**, registered with `Usage = SsoServer`.
2. One or more **SSO user entries** (`Usage = SsoClient`) that map an upstream identity to a
   [security clearance](../../../server/security/authorization/security-clearance-and-permissions.mdx) and database permissions.

Both are managed from **Manage Server → Certificates** in Studio (or through the admin certificates
HTTP API), alongside the cluster's regular server and client certificates.

* In this article:
   * [End-to-end checklist](../../../server/security/sso/configuring-ravendb-for-sso.mdx#end-to-end-checklist)

</Admonition>

<Panel heading="End-to-end checklist">

The full procedure - Studio flow, HTTP payloads, validation rules, supported providers, and the
certificate-renewal caveat - is documented on a single page:

<Admonition type="info" title="See">
[Authentication: SSO Certificates and Users](../../../server/security/authentication/sso-certificates.mdx)
</Admonition>

1. Deploy the SSO application (see [Deploying the SSO Application](../../../server/security/sso/deploying-sso-app.mdx)).
2. As a **Cluster Admin**, [register the SSO server certificate](../../../server/security/authentication/sso-certificates.mdx#register-an-sso-server-certificate) -
   either by fetching it from the SSO URL or by uploading it directly.
3. As an **Operator** or higher, [create SSO user entries](../../../server/security/authentication/sso-certificates.mdx#register-an-sso-user-entry)
   for each user or group, choosing the right provider, identifier, and database permissions.
4. Users now log in at `https://<cluster-alias>.<sso-domain>/studio/`. They authenticate through the
   provider; the SSO server issues a short-lived client certificate carrying their SSO identity; RavenDB
   matches that identity against the registered SSO user entry and applies its permissions.

<Admonition type="warning" title="Renew the SSO server cert with the same key">
SSO server trust is pinned by **public key**. Renewing the SSO certificate with a new key pair changes
the pinning hash and breaks every SSO user entry pinned to the old hash. See
[Renewing the SSO server certificate](../../../server/security/authentication/sso-certificates.mdx#renewing-the-sso-server-certificate)
for the rationale and the key-rotation workaround.
</Admonition>

</Panel>
